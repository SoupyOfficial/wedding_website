## Executive Summary

The Forever Campbells wedding website is in **good overall health**. ## 🔴 Issues & Weak Points

### 1. Failing Test File: `photos-upload.test.ts` (5 failures)

**Severity: Medium** — Blocks CI green if tests were in CI

The mock for `@/lib/providers` exports a mock `getProvider` that returns an `upload` function, but when the test creates mock scenarios for error states (e.g., replacing the provider mid-test with `vi.mocked(getProvider).mockReturnValue(...)`), the underlying service layer (`photo.service.ts`) may be referencing a different instance. Result: all 5 failing tests get HTTP 500 instead of the expected 400/201/503.

**Recommended fix**: Refactor mocks to intercept at the `lib/services/photo.service` level rather than at the provider level, or ensure `getProvider` mock is consistently referenced.

### 2. TypeScript Errors in Test Files (20 errors)

**Severity: Low** — Non-production, but CI runs `tsc --noEmit` which will fail on these

All 20 errors are in `__tests__/` files only:

| File | Errors | Root Cause |
|------|--------|------------|
| `admin-messages.test.ts` | 4 | mock `execute()` return type missing `rows` property |
| `admin-music-actions.test.ts` | 5 | Same as above |
| `apple-music-import.test.ts` | 5 | Mock `PlaylistTrack` missing `album`/`durationMs` + rows issue |
| `weather-forecast.test.tsx` | 1 | Type mismatch on `vi.mocked(WMO_CODE_MAP)` |
| `weather-forecast.test.tsx` | 1 | Missing `afterEach` import (likely `@vitest/run` vs `vitest`) |

**Recommended fix**: Add `rows` to the mock `execute` return type, complete mock objects, fix imports.

### 4. Travel Content Hardcoding (`lib/config/travel-content.ts`)

**Severity: Medium** — 533 lines of hardcoded Orlando/FL venue-specific content

This file contains arrays of airports, theme parks, restaurants, and transportation options — all hardcoded. If the wedding venue changes or information needs updating, developers must touch code rather than admins using the dashboard.

**Recommended**: Move to database with admin-editable interface, or at minimum move to a JSON config file separate from TypeScript logic.

### 5. CI Pipeline Missing Test Execution

**Severity: Medium** — Tests can break without CI awareness

Current CI workflow (`.github/workflows/ci.yml`):
```
1. Lint (ESLint)
2. Type Check (tsc --noEmit)
3. Build (next build)
```

Notably absent: `npm run test` (Vitest) and `npm run test:e2e` (Playwright). The 20 TSC errors in test files would actually be caught since CI runs `tsc --noEmit`, but the 5 runtime test failures would not be.

### 6. Git Dubious Ownership

**Severity: Low** — Environment-specific, not code issue

The repository exists at `C:\Users\JSCam\Documents\Development\wedding_website` but the current execution context runs as `NT AUTHORITY/SYSTEM` while the repo is owned by `SOUPY-DESKTOP/JSCam`. Git operations fail with "dubious ownership" errors. This is a Windows security feature and could block automated git workflows from the current context.

**Fix**: `git config --global --add safe.directory C:/Users/JSCam/Documents/Development/wedding_website`
